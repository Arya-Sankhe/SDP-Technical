# v9.5 Technical Learning Summary

`v9.5.ipynb` keeps the same core MSFT 1-minute probabilistic forecaster from v8.5, but adds a retrieval-augmented pattern memory layer on top of inference. The base model is still a seq2seq attention GRU that predicts the next `50` OHLC return steps; the new idea is to retrieve similar historical windows and blend their realized future paths into each sampled forecast.

## What changed vs v8.5

The forecasting pipeline is still:

- build return-style intraday features
- train a probabilistic seq2seq GRU
- sample multiple future paths autoregressively
- select the best path by trend matching
- evaluate with walk-forward folds and a strictly causal rolling backtest

v9.5 adds a retrieval layer after training:

- encode historical input windows into embeddings
- store each window together with its true future path
- encode the current query window at inference time
- retrieve the top `k=5` nearest historical patterns
- softmax-weight their future paths
- blend that retrieved future with the generated path

In the saved run, the blend is:

`adjusted_path = 0.75 * model_path + 0.25 * retrieved_future`

So this is not a new forecaster architecture. It is an inference-time path adjustment layer on top of the same GRU generator.

## Data and features

The notebook still uses `120` days of `MSFT` 1-minute Alpaca bars, sessionized to regular market hours. In the saved run:

- raw rows from Alpaca: `31,342`
- sessionized rows kept: `30,630`

The feature set goes back to the `10` core columns:

- `rOpen`, `rHigh`, `rLow`, `rClose`
- `logVolChange`, `logTradeCountChange`
- `vwapDelta`
- `rangeFrac`
- `orderFlowProxy`
- `tickPressure`

Targets are still the next `50` steps of:

- `rOpen`, `rHigh`, `rLow`, `rClose`

As in earlier versions, the window builder appends an imputation-fraction channel, so the effective model input width is `11` even though `feature_dim = 10`.

Rows are also tagged with:

- `row_imputed`
- `row_open_skip`
- `prev_close`

Those flags are used to drop contaminated target windows and skip the first `6` bars after the open.

## Core model

The forecaster itself is still `Seq2SeqAttnGRU`:

- encoder: `GRU(input_size=11, hidden_size=256, num_layers=2, dropout=0.20)`
- decoder: autoregressive `GRUCell`
- attention: dot-product attention over encoder states
- heads: one predicts `mu`, one predicts `log_sigma` for each OHLC return

So the model still learns a Gaussian distribution over future returns, not just a point estimate.

Training setup in v9.5 is unchanged from the earlier probabilistic versions:

- `AdamW(lr=5e-4, weight_decay=1e-5)`
- `BATCH_SIZE = 256`
- teacher forcing decays by `0.95` each epoch
- `ReduceLROnPlateau(factor=0.5, patience=3, min_lr=1e-6)`
- gradient clipping at `1.0`

Loss:

`total = weighted_nll + 0.3 * range_loss + 0.5 * volatility_match_loss + 0.1 * directional_penalty`

Later horizon steps are upweighted with:

`1 + (t / (H - 1)) ^ 1.5`

## RAG pattern memory

This is the main technical addition.

The retriever has three pieces:

1. `PatternEncoder`
2. `PatternDatabase`
3. `RAGPatternRetriever`

`PatternEncoder` is a small LSTM encoder:

- hidden size: `128`
- layers: `2`
- embedding dimension: `64`
- projection: `Linear -> LayerNorm -> Tanh`

`PatternDatabase` stores:

- normalized embeddings
- original input sequences
- true future paths

Search is cosine-similarity top-k retrieval. The notebook tries to import `faiss`, but the database is always built with `use_faiss=False`, so the saved implementation actually uses brute-force torch matmul search, not a FAISS index.

At inference time, retrieval works like this:

1. encode the current input window
2. retrieve the top `5` historical windows
3. softmax their similarity scores
4. compute a weighted average of their future paths
5. blend that average into the sampled forecast with weight `0.25`

Important implementation nuance: the retriever is attached only inside `generate_realistic()`. The deterministic forward pass `model(x)` does not use retrieval.

That means:

- ensemble path generation uses RAG
- trend-selected visualization paths use RAG
- rolling backtest path generation uses RAG
- one-step MAE metrics do **not** use RAG
- the lookback sweep score also does **not** use RAG

## Important caveat in the notebook code

The retriever encoder is never trained.

In both walk-forward and rolling mode, the notebook does:

- train the GRU forecaster
- instantiate a fresh `RAGPatternRetriever`
- build the retrieval database from historical windows
- attach the retriever to the already-trained model

But that retriever is not part of the optimizer and is never trained with a retrieval loss or joint objective. So in the saved notebook, the embedding space is produced by a newly initialized LSTM encoder, not a learned pattern encoder. Retrieval still works mechanically, but similarity is being measured in an untrained embedding space.

If you reimplement this technique seriously, that is the first thing I would fix.

## Main takeaway

v9.5 is best understood as:

- the same probabilistic attention-GRU forecaster from v8.5
- the same trend-matched ensemble path selection
- plus a retrieval layer that blends similar historical future paths into generated forecasts

If you want to reproduce the technique, the pieces that matter most are:

- the seq2seq probabilistic GRU forecaster
- stochastic path generation with temperature scaling
- trend-based path selection
- a database of historical windows paired with their realized future paths
- top-k similarity retrieval and weighted path blending

But if you want the RAG part to be genuinely meaningful, the retriever encoder needs to be trained rather than freshly initialized after the forecaster finishes.
