# v9.2 Technical Learning Summary

`v9.2.ipynb` is the Phase 2 extension of the same v8.5/v9.x GRU forecasting family. It does **not** introduce a new neural architecture. Instead, it adds a market-regime subsystem based on turbulence and ATR, and uses that subsystem to adjust inference temperature during the strictly causal rolling backtest.

## What changed vs v9.1

v9.1 expanded the feature space heavily with technical indicators. v9.2 moves in the opposite direction: it reduces the engineered feature block to `15` features total, made up of the original `10` core features plus `5` regime-related features.

Saved run summary:

- `LOOKBACK_DAYS = 120`
- `HORIZON = 50`
- `feature_dim = 15`
- `regime_feature_count = 5`
- sessionized rows kept: `31,410`
- sessions kept: `81 / 84`

During windowing, the notebook still appends one constant imputation-fraction channel, so the actual model input width is `16`, not `15`.

## Feature set

Core features preserved:

- `rOpen`, `rHigh`, `rLow`, `rClose`
- `logVolChange`, `logTradeCountChange`
- `vwapDelta`
- `rangeFrac`
- `orderFlowProxy`
- `tickPressure`

New regime-related features:

- `atr_14`
- `atr_14_pct`
- `returns`
- `turbulence_60`
- `regime_indicator`

Important implementation detail: `regime_indicator` is not dynamically computed in the feature frame. It is hard-coded as `1.0` for every row. So the actual regime awareness in v9.2 comes mostly from:

- `atr_14` and `atr_14_pct`
- `returns`
- `turbulence_60`
- a separate rolling-time regime detector that adjusts inference temperature

It does **not** train on a changing categorical regime label.

## Regime detection system

The notebook introduces three regimes:

- `NORMAL`
- `ELEVATED`
- `CRISIS`

The regime detector combines:

- a turbulence index
- current ATR percentile

Turbulence is implemented as a Kritzman-Li style Mahalanobis-distance score over recent returns:

`d_t = sqrt((r_t - mu)^T Sigma^{-1} (r_t - mu))`

Configuration used:

- turbulence / ATR lookback: `60`
- normal threshold: `75th percentile`
- elevated threshold: `90th percentile`

Temperature multipliers by regime:

- `NORMAL -> 1.0`
- `ELEVATED -> 1.3`
- `CRISIS -> 1.8`

Position-size multipliers are also defined in config:

- `NORMAL -> 1.0`
- `ELEVATED -> 0.7`
- `CRISIS -> 0.3`

but they are not used in model training or forecasting logic here. They are just part of the regime framework.

## Model and training

The model is still `Seq2SeqAttnGRU`:

- encoder GRU with `hidden_size=256`, `num_layers=2`, `dropout=0.20`
- autoregressive `GRUCell` decoder
- attention over encoder states
- separate `mu` and `log_sigma` heads for probabilistic OHLC return prediction

Training is also unchanged:

- `AdamW`, learning rate `5e-4`, weight decay `1e-5`
- batch size `256`
- teacher forcing decays from `1.0` with factor `0.95`
- `ReduceLROnPlateau(factor=0.5, patience=3, min_lr=1e-6)`
- gradient clipping at `1.0`

Loss:

`total = NLL + 0.3 * range_loss + 0.5 * volatility_match_loss + 0.1 * directional_penalty`

Additional details:

- inputs are standardized using train-only statistics
- targets are not standardized
- later forecast steps are upweighted with `1 + (t / (H-1))^1.5`
- target windows are dropped if they contain imputed rows or the first `6` minutes after the open

## Inference logic

The non-rolling inference path is still the same ensemble trend-selection method from v7.5/v8.5:

- `ENSEMBLE_SIZE = 20`
- `SAMPLING_TEMPERATURE = 1.5`
- `TREND_LOOKBACK_BARS = 20`
- `STRONG_TREND_THRESHOLD = 0.002`
- `MIN_PREDICTED_VOL = 0.0001`

`generate_realistic()` samples recursively from the predicted Gaussian and feeds sampled returns back into the decoder. For the first `5` steps it can override sigma using recent realized volatility. After converting returns back to prices, candle validity is enforced so highs and lows remain structurally valid.

Path selection is still momentum matching:

1. compute historical slope from the last `20` context closes
2. generate `20` sampled paths
3. compute each path slope
4. optionally reject opposite-direction paths in strong-trend cases
5. pick the path whose slope is closest to the historical slope

## Rolling backtest behavior

The rolling backtest is where v9.2 becomes regime-aware.

At each prediction anchor:

1. the backtester observes only `[t-window, ..., t-1]`
2. it reads the current return and ATR at time `t`
3. it classifies the current regime with `MarketRegimeDetector`
4. it multiplies the base temperature by the regime multiplier
5. it generates the sampled path using the adjusted temperature

So regime detection affects **sampling temperature**, not the network weights.

Base rolling configuration:

- base rolling temperature: `1.5`
- intraday schedule: `1.25` from `09:30-10:15`, `1.45` from `10:15-14:00`, `1.60` from `14:00-16:00`
- rolling date selected in saved run: `2026-04-01`

Important implementation nuance: like v9.1, the rolling backtest does **not** use the sweep-selected lookback. It retrains a separate rolling model with `DEFAULT_LOOKBACK = 96`, because cells `56-58` call the rolling trainer and backtester with `DEFAULT_LOOKBACK`, not the sweep-selected `256`.

Saved rolling regime distribution:

- elevated: `117` predictions (`30.0%`)
- crisis: `13` predictions (`3.33%`)
- regime transitions: `117`

## Observed results and caveats

Walk-forward results with the selected `256`-bar lookback:

- `slice_1` model close MAE: `0.3008` vs persistence `0.2128`
- `slice_2` model close MAE: `0.2499` vs persistence `0.1800`

Directional accuracy:

- `slice_1`: `50.11%` vs persistence `74.06%`
- `slice_2`: `51.18%` vs persistence `73.50%`

This means the added regime framework still did not beat the persistence baseline in the saved walk-forward run.

There are also two important caveats from the notebook itself:

1. The regime-aware logic is mostly an inference-time wrapper. The GRU architecture and core loss are unchanged.
2. The synthetic Phase 2 validation cell shows turbulence rising during crisis-like data, but the sample `MarketRegimeDetector` test still ends in `ELEVATED`, not `CRISIS`, with `0` logged regime transitions. So the framework exists, but the calibration / test harness does not yet strongly demonstrate robust crisis detection.

The main implementation takeaway is that v9.2 is best understood as:

- the same probabilistic attention-GRU forecaster
- a smaller feature set centered on volatility and turbulence
- a regime detector wrapped around rolling inference
- regime-aware temperature scaling during causal backtesting

If you were reimplementing it, the essential pieces are the turbulence calculator, the regime detector thresholds and multipliers, the unchanged GRU path generator, and the rolling-time temperature adjustment based on detected regime.
