# v9.1 Technical Learning Summary

`v9.1.ipynb` is the Phase 1 extension of the v8.5 pipeline. It keeps the same probabilistic seq2seq GRU, ensemble path generation, trend-based path selection, and strictly causal rolling backtest, but expands the input representation with a dedicated technical-indicator feature block.

## What changed vs v8.5

The main change is feature engineering. v8.5 used `10` core microstructure/return features; v9.1 adds `32` technical-indicator features, bringing the engineered feature count to `42`. During window creation, one more constant channel is appended for the fraction of imputed bars in the lookback window, so the model actually receives `43` input channels per timestep.

In the saved run, the notebook fetched `120` days of `MSFT` 1-minute data, kept `81 / 84` exchange-calendar sessions, and ended with `31,410` usable rows after session filtering.

## Model and training

The architecture is unchanged from v8.5:

- Encoder: `GRU(hidden_size=256, num_layers=2, dropout=0.20)`
- Decoder: autoregressive `GRUCell`
- Attention: learned projection plus attention over encoder states
- Output heads: one head predicts `mu`, one predicts `log_sigma` for each OHLC return

This is a probabilistic forecaster, not a deterministic regression model. The decoder predicts a Gaussian distribution at each future step, and inference samples from that distribution recursively.

Training setup:

- `AdamW`, learning rate `5e-4`, weight decay `1e-5`
- Batch size `256`
- Teacher forcing starts at `1.0` and decays by `0.95` per epoch
- `ReduceLROnPlateau(factor=0.5, patience=3, min_lr=1e-6)`
- Gradient clipping at `1.0`

Loss:

`total = NLL + 0.3 * range_loss + 0.5 * volatility_match_loss + 0.1 * directional_penalty`

Important details:

- Inputs are standardized with train-only statistics.
- Targets are not standardized.
- Later forecast steps are upweighted with `1 + (t / (H-1))^1.5`.
- Windows are dropped if the target horizon contains imputed bars or the first `6` bars after the session open.

## Inference logic

v9.1 keeps the same ensemble trend-selection logic as v8.5:

- `ENSEMBLE_SIZE = 20`
- `SAMPLING_TEMPERATURE = 1.5`
- `TREND_LOOKBACK_BARS = 20`
- `STRONG_TREND_THRESHOLD = 0.002`
- `MIN_PREDICTED_VOL = 0.0001`

`generate_realistic()` samples recursively from `mu + noise * sigma`, where `sigma = exp(log_sigma) * temperature`. For the first `5` forecast steps, sigma can be replaced by recent realized volatility from the context window. After sampling, OHLC candles are repaired so `High >= max(Open, Close)` and `Low <= min(Open, Close)`.

Final path selection works like this:

1. Compute the recent historical trend slope from the last `20` close prices in the input context.
2. Generate `20` stochastic future paths using different seeds.
3. Compute each path slope from predicted close returns.
4. If the historical slope is strong, reject paths going in the opposite direction.
5. Select the path whose slope is closest to the historical slope.

## Rolling backtest implementation

The rolling engine is still strictly causal:

- Prediction anchors run every minute from `09:30` to `16:00`
- At anchor time `t`, the visible context is exactly `[t-window, ..., t-1]`
- The scaler for rolling mode is fit only on rows before the selected backtest date
- Only windows whose full target horizon ends before the backtest date are used for rolling-model training

Intraday temperature schedule:

- `09:30-10:15 -> 1.25`
- `10:15-14:00 -> 1.45`
- `14:00-16:00 -> 1.60`

The saved rolling date was:

- `2026-04-01`

Important implementation nuance: the rolling backtest does **not** use the sweep-selected lookback of `160`. It retrains a separate rolling model with `DEFAULT_LOOKBACK = 96`, because cells `58-60` call `train_v7_model_for_rolling(... window=DEFAULT_LOOKBACK)` and `runrollingbacktest(... windowsize=DEFAULT_LOOKBACK)`.

The notebook also asserts:

- first predicted timestamp equals anchor timestamp
- context ends strictly at `t-1`
- number of produced predictions equals the expected number of minute anchors

In the saved run it generated `390` rolling predictions and saved `390` standalone frames.

## Observed results and caveats

Full walk-forward results with the selected `160`-bar lookback:

- `slice_1` model close MAE: `0.3001` vs persistence `0.2128`
- `slice_2` model close MAE: `0.2506` vs persistence `0.1800`

Directional accuracy:

- `slice_1`: `50.42%` vs persistence `74.06%`
- `slice_2`: `51.26%` vs persistence `73.50%`

Rolling backtest summary:

- directional hit rate at `t+1`: `49.74%`
- path MAE @ step 1: `$0.9241`
- path MAE @ step 5: `$4.0010`
- path MAE @ step 10: `$29.0916`
- path MAE @ step 15: `$15.1613`
- trend correlation: `-0.0189`

So the main technical takeaway is that v9.1 is not a new architecture. It is v8.5 plus a large technical-indicator feature block. The parts that matter for reimplementation are:

- feature construction with the `42` engineered inputs
- probabilistic attention-GRU sequence model
- stochastic autoregressive path sampling
- trend-matched ensemble selection
- strictly causal rolling evaluation

The saved metrics suggest the additional indicators did not yet make the model outperform a simple persistence baseline, and the rolling-path errors become unstable at longer horizons.
