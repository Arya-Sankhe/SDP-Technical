# v9.3 Technical Learning Summary

`v9.3.ipynb` is the Phase 3 version of the same MSFT 1-minute forecasting pipeline, but this time the encoder is no longer pure GRU. The main change is a hybrid encoder that combines an iTransformer-style feature-attention encoder with a GRU encoder, while keeping the same probabilistic decoder, ensemble path generation, and strictly causal rolling backtest setup.

## What changed vs v9.2

The core forecasting workflow is still the same:

- predict OHLC return distributions
- sample autoregressive future paths
- select one sampled path by trend matching
- evaluate with walk-forward folds and a rolling causal backtest

What changes in v9.3 is the encoder architecture and the training/runtime policy around GPU memory.

## Hybrid iTransformer encoder

This is the main technical addition.

The encoder has three pieces:

1. `iTransformerEncoder`
2. GRU encoder
3. fusion MLP

### iTransformer side

The iTransformer path does:

- transpose input from `[batch, time, features]` to `[batch, features, time, 1]`
- project each scalar feature stream with `Linear(1 -> d_model)`
- add sinusoidal positional encoding over time
- run attention **across variables/features**, not across time
- average over time
- flatten to `[batch, features * d_model]`

Configuration:

- `D_MODEL = 128`
- `N_HEADS = 8`
- `N_LAYERS = 2`

### GRU side

In parallel, a standard GRU encoder runs over the original input sequence and produces:

- full sequence states `gru_out`
- final hidden state `gru_last`

### Fusion

The final fused encoder representation is:

`concat(iTransformer_flat, gru_last) -> MLP -> LayerNorm`

That fused vector initializes the decoder hidden state, while the GRU sequence output is still used as the attention memory for decoding.

So the decoder is still GRU-based, but its starting state now mixes:

- cross-feature transformer information
- sequential GRU information

## Decoder and probabilistic output

The decoder remains the same general idea as earlier versions:

- one `GRUCell` rolled forward autoregressively
- attention over encoder sequence states
- one head for `mu`
- one head for `log_sigma`

This means the model still forecasts a Gaussian distribution over OHLC returns at every future step instead of only point estimates.

## Training recipe

Base training remains similar:

- `AdamW`, learning rate `5e-4`, weight decay `1e-5`
- teacher forcing decays from `1.0` with factor `0.95`
- `ReduceLROnPlateau(factor=0.5, patience=3, min_lr=1e-6)`
- gradient clipping at `1.0`

Loss:

`total = NLL + 0.3 * range_loss + 0.5 * volatility_match_loss + 0.1 * directional_penalty`

Important v9.3 implementation details:

- automatic mixed precision is enabled on CUDA
- `GradScaler` is used during training
- explicit memory cleanup is added via `gc.collect()`, `torch.cuda.empty_cache()`, and `torch.cuda.ipc_collect()`
- prediction on large test sets is done in batches with `predict_mean_in_batches()`

This notebook is clearly optimized to make the larger hybrid model run on limited GPU memory.

## Inference logic

The path-generation logic is still the same family as v8.5:

- `SAMPLING_TEMPERATURE = 1.5`
- `ENSEMBLE_SIZE = 8` in this saved run because low-VRAM mode was active
- `TREND_LOOKBACK_BARS = 20`
- `STRONG_TREND_THRESHOLD = 0.002`
- `MIN_PREDICTED_VOL = 0.0001`

At inference time:

1. generate multiple stochastic paths with different seeds
2. compute slope of each path
3. compare to the recent historical slope
4. choose the path whose slope is closest to the recent trend

The hybrid encoder changes the representation that drives forecasting, but not the ensemble selection logic.

## Rolling backtest

The rolling engine is still strictly causal and structurally unchanged from the earlier causal notebooks:

- predict every minute from `09:30` to `16:00`
- use only `[t-window, ..., t-1]` as visible context at anchor time `t`
- fit the scaler only on rows before the backtest date
- keep only windows whose full target horizon ends before the backtest date

## Main takeaway

v9.3 is best understood as:

- the same probabilistic multi-step OHLC forecaster
- the same ensemble trend-selection inference logic
- a new hybrid encoder that combines iTransformer feature attention with GRU sequence encoding
- a memory-aware training implementation designed for an `8 GB` GPU

If you were reimplementing it, the parts that matter most are:

- the inverted-transformer encoder path with feature-wise attention
- the GRU encoder path and fusion block
- AMP plus batched inference plus memory cleanup for low-VRAM training
- the unchanged probabilistic decoder and causal rolling evaluation loop
